提示词写得好不好，差别往往在于模型是否清楚知道「要做什么、按什么格式输出、哪些内容不能编造」。

## 三要素模板

每次写 Prompt 至少包含：

1. **角色**：你希望模型扮演什么身份（例如：资深 Python 工程师）
2. **任务**：具体要完成什么（例如：把以下 JSON 转成 Markdown 表格）
3. **输出格式**：列点、表格、JSON、字数上限等

示例：

```
你是资深技术文档编辑。请把以下 API 错误说明改写为简体中文 FAQ。
要求：
- 使用 ## 作为每题标题
- 每题回答不超过 80 字
- 不确定的内容标注「需官方确认」
```

## 减少幻觉的约束

- 要求「仅根据提供的材料回答，材料不足时明确说不知道」
- 需要引用时，指定「逐条对应原文段落」
- 对数字、日期、法规类内容，要求标注来源或写「待核实」

## Few-Shot 示例

给 1–2 个输入输出示例，比长篇描述更有效。例如做分类任务：

```
输入：登录时出现 503
输出：{"type":"server","action":"retry_later"}

输入：API 返回 401
输出：{"type":"auth","action":"check_api_key"}
```

## 常见问题与修正

| 问题 | 原因 | 修正 |
|------|------|------|
| 回答太长 | 未限制字数 | 加入「不超过 N 字」 |
| 格式混乱 | 未指定格式 | 要求 JSON / Markdown 模板 |
| 自行补充不存在的功能 | 缺少边界 | 加入「不得臆测未提供的信息」 |

## 迭代方式

1. 先用短 Prompt 测试能否完成任务
2. 看输出差异，再补充约束
3. 固定可用的 Prompt 存成模板，避免每次从零写起
