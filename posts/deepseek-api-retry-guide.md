503 与 429 都是暂时性错误，但处理方式不同：503 通常表示服务端暂时不可用，429 表示你的请求过快。两者都需要重试，但不能无脑循环。

## 错误区分

| 状态码 | 含义 | 是否适合重试 |
|--------|------|--------------|
| 503 | 服务暂时不可用 | 是，需退避 |
| 429 | 超出速率限制 | 是，需更长等待 |
| 401/403 | 认证或权限问题 | 否，检查 Key |
| 400 | 请求格式错误 | 否，修正参数 |

## 指数退避重试（Exponential Backoff）

基本流程：

1. 首次失败后等待 1 秒
2. 第二次等待 2 秒，第三次 4 秒
3. 加入随机抖动（jitter），避免多个客户端同时重试
4. 设定 `maxRetries = 3` 或 `5`，超过则返回友好错误

## 熔断器（Circuit Breaker）

连续失败达到阈值时，暂停请求一段时间，避免把已过载的服务打得更忙。

典型参数：

- 失败阈值：5 次连续 503/429
- 熔断开启时间：30–60 秒
- 半开状态：允许 1 个探测请求，成功则恢复

## 429 的特殊处理

- 优先读取 `Retry-After` 响应头
- 若无该字段，429 的等待时间应比 503 更长
- 对批量任务使用队列 + 固定并发（如同时最多 2 个请求）

## 用户端应有的降级方案

- 返回「服务繁忙，请稍后再试」而非 raw error
- 对可缓存问题（FAQ、固定模板）使用本地缓存
- 关键流程提供人工兜底入口

## 排查日志关键字段

记录 HTTP 状态码、重试次数、请求耗时、使用的模型与 Token 数，方便定位是平台高峰还是自身请求模式问题。
